Vid — Foundational Design

A code-review provenance system: which units of code a human (or a tool) reviewed — surviving refactors, resisting forgery, across many languages.

Status: foundation locked, pre-build · implementation Rust · greenfield · brand Vidi Vouch, tool vid · pressure-tested by a 10-agent adversarial review (verdict: sound-with-changes) · all 7 open decisions settled · 2026-06-18; editor-integration & GitHub surfaces refined 2026-06-19 (mockups in editor-plugin-designs.html).

What a review attests — and what it does not A vid review asserts that a named identity reviewed a unit of code at a specific normalized shape. It is not proof of understanding, nor of behavioral correctness, nor (in phase 1) cryptographic proof of who. The headline deliverable is the "un-reviewed by a human" report; everything below exists to make that report mean something.

1. Purpose & headline deliverable

vid records which code units a human reviewed, in a way that survives refactoring and resists forgery, and produces an "un-reviewed (by a human) report." It began as tooling for a large LLM-assisted port (legacy → rebuild), where reviewers needed to mark code "read and verified equivalent" and see what remained unchecked.

A review attaches to a code unit (function, class, module statement, doc section), not to a diff or a line. The signal is durable: a review holds through cosmetic change and rename/move, and goes stale on behavioral change — with no dependence on git history (rebase-, squash-, shallow-clone-safe).

2. Threat model & trust posture

  • Phase 1 is an honesty / UX nudge, not unforgeability. Identity is self-asserted, the human/tool boundary lives in a bypassable hook, and the store is a committed file. A one-line append forges "human coverage." vid makes such forgery diff-visible and CI-checkable — it does not prevent it.
  • External prerequisites without which human coverage is theater: a protected branch, a required vid verify check, and CODEOWNERS on the policy file. vid assumes these; it does not provide them.
  • Phase 2 binds trust to cryptography (§11): the lane becomes a property of the signing credential issuer; records are DSSE-signed and logged.
  • Irreducible residual risk: an insider (or in-house agent) holding a live human session can sign as that human even in phase 2. Documented, not solved, without per-review hardware attestation.
ConsequenceThe headline report prints a "human coverage is self-asserted and unverified" banner until signing lands, and phase-1 human-lane records are excluded from any senior/release gate.

3. The core primitive

A reviewable unit is a node in a per-language normalized tree produced by a versioned NormalizationProfile. Each node carries two independent digests. This primitive is implementation-language-agnostic — it is an algorithm and a data model, equally expressible in Rust (our choice) as in any language.

DigestWhat it isRole
Exact hashsha256 of the canonical alpha-normalized serialization, parameterized by (profileVersion, scopeModelVersion); free-name binding-kind sensitivity is planned (not in v1, §4/§14)Identity + freshness. Strictly binary (matches / STALE). The only thing that gates re-review and human coverage.
Similarity digestthe multiset of the node's children's exact hashes, stored in the record at review timeSeverity only — advisory. Never gates anything; drives queue triage and gutter intensity.

Severity

severity = bucket( max(relative, absolute) ), relative = 1 − RuzickaMultisetJaccard(stored_children, current_children), absolute = count of added/removed/modified child hashes. A single changed child can never bucket below MINOR — a one-line auth flip in a large function cannot hide as NONE.

Forced change · resolvedJaccard is true multiset Ruzicka (sum-of-mins / sum-of-maxes), with a mandated unit test that {s,s} vs {s} is nonzero. Set-based is forbidden.

Ordered vs unordered children

A per-node-kind flag drives both digests, so the signals can't disagree. Ordered (function/method bodies, control-flow bodies, import prologue): positional exact hash, adjacent-pair bigram shingles for similarity, so reorders surface nonzero severity. Unordered (module top-level, most class members, HTML attributes): sorted child-hashes, unigram multiset — reorder invariant.

Forced change · resolvedClass-body ordering is conditional: ordered when a field-ordering base/decorator (dataclass/Pydantic/NamedTuple/Enum), a sibling-referencing member decorator (@x.setter), or @overload is present; unordered otherwise. A reorder of a conditionally-ordered class reads full STALE (decision #6); the dual-digest reserves a later "reorder-only" variant with no format change.

"Whole file" is not a hashable unit — file coverage is derived from child-unit coverage, so appending a sibling never stales the file.

flowchart TD SRC[Source file, any language] --> PROF[NormalizationProfile - tree-sitter + scope binder] PROF --> TREE[Normalized node tree] TREE --> EX["Exact hash = sha256(canonical)\nbinary freshness"] TREE --> SIM["Child-hash multiset\nadvisory severity"] EX --> REC[in-toto Statement record] SIM --> REC REC --> STORE[".vid/reviews.jsonl\nunion-merge, positive claims only"] STORE --> VERIFY[vid verify] STORE --> REPORT[un-reviewed report]

4. Alpha-normalization & the scope model

Alpha-normalization rewrites a unit's own name, parameters, and locally-bound names to positional placeholders, keeping free names (called functions, imports, attributes, operators, types). Invariant to rename / param-rename / local-rename / move / reformat; sensitive to literal / called-name / added-param / decorator / control-flow change.

Parser-agnostic by designBound/free classification runs against vid's own versioned canonical scope model, built on the parse tree — never the parser's built-in scope and never a language's live interpreter. This decision was always parser-agnostic, so it transfers unchanged to Rust + tree-sitter: the parser provides the tree; the profile provides the bindings via a hand-written CST binder keyed on node type. scopeModelVersion is a signed field.
Planned · not in v1The exact hash is designed to become sensitive to free-name binding-kind transitions (imported ↔ local-shadow ↔ builtin), closing a lift-and-shift vector where a kept free name is later shadowed/monkeypatched. The shipped Rust profile does not yet implement binding-kind sensitivity (a known v1 simplification, §14): a local rename that shadows a previously-free name reads STALE — the safe over-report direction. The hash attests to normalized syntactic shape, not semantics.

5. Languages & NormalizationProfiles

vid is built to be versatile across many languages. The architecture: tree-sitter as the universal frontend — one grammar crate per language (≈any language) — behind a single NormalizationProfile trait. Adding a language = add a grammar crate + write its profile. A profile declares: unit node-types, bound/free scope rules (CST binder), ordered/unordered per node-kind, ignore-for-hash, and semantic toggles.

Dedicated-parser escape hatchThe default substrate is uniform tree-sitter. Where a language needs higher fidelity than its grammar gives, a dedicated parser can be swapped in behind the same trait (e.g. ruff_python_ast for Python) without touching the engine. tree-sitter is a CST (we walk named nodes) and may differ from a language's official AST in corners — fine for a hashing/identity tool, since we define the canonical form over the chosen parser and version it with profileVersion.

MVP languages & their tier

TierLanguages (MVP)Treatment
BindingPython, JavaScript, Rustfull alpha-normalization; own CST scope binder per language (the binder pattern generalizes across all three)
StructuralHTML (+ Django/Jinja templates)no bindings → structural hash of a spec tree; coarse unit boundary (landmark/block/section); <script>/<style> injections routed to their profiles
Prose + fencesMarkdown (CommonMark/GFM)comrak container; prose sections (heading-scoped, weak invariance, sentence-shingle severity); code fences routed by info-string, fragment-tolerant + opaque-text fallback

vid reviews vid: the Rust profile (tree-sitter-rust) means the tool self-hosts — its own codebase is a first-class review target.

6. Move / rename / extract reconcile

A review re-anchors to a moved/renamed unit only when all hold: old address gone, exact hash unique repo-globally, and unit above the complexity floor. It never silently transfers a human verdict and never reuses the old signature — it emits a new, unsigned "review may have moved here" suggestion requiring one-keystroke human re-confirm.

Why the complexity floorEmpty/trivial units (e.g. an empty __init__.py, mod.rs re-exports, trivial passthroughs) collide on hash, so a naive uniqueness rule churns or launders a human vouch onto attacker code. Sub-floor units are structurally ineligible for auto-reconcile.
Accepted limitationExtract-method / cross-function move loses the review. Mitigated by the same non-authoritative suggestion (diff old child-multiset against nearby new units); never auto-attached.

7. The record

Each record is an in-toto Statement, DSSE-envelope-ready from day one, validated against the in-repo JSON Schemas in schemas/ (the source of truth — human-review-v1 and tool-review-v1); predicateType is an opaque id whose URL need not resolve.

{
  "_type": "https://in-toto.io/Statement/v1",
  "subject": [{
    "name": "repo+path::qualpath",
    "digest": { "sha256": "<hex of canonical alpha-normalized form>" }   // ONLY a true content digest
  }],
  "predicateType": "https://vidivouch.com/vid/human-review/v1",            // or .../tool-review/v1
  "predicate": {
    "criteria": ["equivalent"],          // CLOSED core vocab + namespaced extensions; verify fails closed on unknown
    "rigor": "read-fully",               // skimmed | read-fully | ran-tests
    "attesterIdentity": { },             // person (human lane) | tool+model+version+harness (tool lane)
    "verdict": "approved",
    "timestamp": "2026-06-18T...",       // ADVISORY; precedence uses Rekor/git topology, never this
    "similarityMultiset": [ ],           // child-hash collection at review time (drives severity)
    "profileVersion": "py-1", "scopeModelVersion": "py-scope-1",
    "invarianceGrade": "code",           // code | structural | opaque
    "signatureState": "self-asserted"    // phase-1 ships self-asserted only; signed-lenient | signed-verified are phase-2 (unshipped)
  }
}
in-toto conformancesubject.digest is a true content digest only ({sha256}). The similarity multiset + version fields live in the predicate — a DSSE signature must never freeze a value the engine compares fuzzily.
Path confidentiality — settled 2026-06-19subject.name's repo+path::qualpath is the one field that names files and symbols in plaintext. Anything crossing the repo trust boundary (the Rekor log, a shared/aggregated report) carries an opaque path token — a repo-keyed encryption of the qualpath (deterministic HMAC is the simpler fallback) — never the plaintext. The plaintext stays an in-repo-only display annotation, outside the DSSE envelope (same trust boundary as the code → zero marginal leak; retained so a deleted path still renders for "moved-from" display). Move/rename/orphan detection is unaffected: §6 matches by content digest, not path, and the stored token preserves the old address even after the path is gone; the §11 anti-copy binding still holds (a copy elsewhere → different token). Key is repo-scoped — committed in .vid/ for private repos (keeps the public log opaque), a no-op for public repos where paths aren't secret. Residual: deterministic tokens reveal structural equality in a shared log, and cross-repo moves would need a shared key — both out of scope for now.

8. Two lanes & the class boundary

  • human-review — a person vouched. The only thing that counts toward human coverage.
  • tool-review — a tool/model examined it. Informational; never counts as human coverage; a bot can't turn a unit green.
Forced change · resolvedThe boundary is bound to the signing-credential issuer, not a self-asserted string: human-review requires an interactive-human OIDC issuer; tool-review requires a machine workload identity that can't obtain the human audience. Until signing lands, a committed identity allowlist gates the human lane; verify flags any human-lane record from a bot/CI committer or off-allowlist attester. The boundary cannot be a write-time hook (bypassable).

9. Monotonicity, storage & revocation

Store: committed .vid/reviews.jsonl + a .gitattributes merge=union rule — one Statement per line, so concurrent reviews of different units merge conflict-free. The merged file is untrusted input that must pass schema + signature verification; signing is per-line DSSE (file-level signing is incompatible with union-merge).

Monotonicity is enforced, not asserted
  • The union file holds positive monotone claims only; absence means "no claim." Policy may only be a positive function of the record set.
  • The shipped ledger carries only reviews + REVOKEs. A separate negative "found-issues" findings ledger is not shipped (a possible later extension).
  • Revocation is a positive REVOKE referencing the target's content hash, with revoke-wins precedence.
  • trust.toml requirements are monotone by construction: an accept row references only >= minimums — there is no negation and no way to reference record absence, so absence is unrepresentable and more/stronger reviews can only flip a unit from fail to pass. vid verify dedupes content-identical lines.
Decision #5 — settledUnion-merge precedence: revoke-wins, ordered by Rekor inclusion time (phase 2) or git topology — never the self-asserted timestamp.

10. trust.toml — the policy TCB

.vid/trust.toml is separate from the facts and is the trusted computing base root: an accept-list policy over two unmerged axes — human role (intern/junior/senior/principal) and a single machine ladder of AI models and 3rd-party tools — plus a [reviewers] identity allowlist, severity thresholds, and cargo-vet-style exemptions. It is applied at report/verify time, so the bar is re-scorable without rewriting history.

GovernanceA protected path: CODEOWNERS-gated, author-cannot-approve-own-change, no same-PR self-promotion or self-exemption. The [reviewers] allowlist is the phase-1 anti-forgery gate: a human review whose id is not listed is recorded but never counted (surfaced as unverified).

The accept-list schema (decision #3)

Because the two axes are never merged into one rank, a requirement is just a set of points on the (human, machine) grid — an accept list. Each [[scope]] binds a path glob to an accept list; a row is one point (human >= …, machine >= …). Any row passing satisfies the scope (the list is a disjunction); within a row, every field is a >= minimum that must hold together (a conjunction). Where two scopes overlap, both must pass — overlap is conjunctive and can only raise the bar.

[axes]
human.order   = ["intern", "junior", "senior", "principal"]   # ranked, ascending
machine.order = ["coderabbit", "cubic", "claude-sonnet-4-6", "claude-opus-4-8"]  # tools AND models, one ladder

[reviewers]                       # the human-identity allowlist (off-list reviews never count)
"alice@acme.com" = "principal"
"bob@acme.com"   = "intern"

[[scope]]                         # billing needs a principal AND opus (one row, two minimums)
path   = "src/billing/**"
accept = [ { human = "principal", machine = "claude-opus-4-8" } ]

[[scope]]                         # dashboards: any of these rows passes (a disjunction)
path   = "src/dashboards/**"
accept = [ { human = "intern" }, { machine = "coderabbit" } ]

default = [ { human = "intern" } ]   # the bar for paths matched by no scope (else: any known human)

[[exemptions]]                    # cargo-vet-style: deliberately excluded from "unreviewed"
unit   = "repo+legacy.rs::shim"
reason = "audited out-of-band; deprecated"

[severity]
major = 0.25                      # advisory only — minor < 25% changed · major >= 25%

Requirements are monotone by construction: an accept row references only >= minimums, with no negation and no way to reference record absence — there is no expression parser and nothing to "reject," so absence is unrepresentable and more/stronger reviews can only flip a unit from fail to pass. Redefining "covered" (editing axes, reviewers, scopes, or exemptions) is a CODEOWNERS-gated change.

11. Signing seam

Phase 2 = DSSE + Sigstore/gitsign (keyless OIDC → Fulcio short-lived cert → Rekor transparency log). The human/machine boundary is enforced by separate IdP audiences. The DSSE subject binds (repo, path, hash, profileVersion, scopeModelVersion) so a verbatim copy into another module doesn't carry the signature.

Decision #7 — settledShip the signing envelope in the record from day one with lenient phase-1 verification (with an explicit signatureState + the unverified banner so "has envelope" ≠ "is secure"); flip enforcement on at phase 2. Retrofitting the envelope after data exists is the migration the greenfield reset avoids.

Operability: public Rekor leaks emails/repo names (qualpaths are already opaque per §7) and GitHub shows no "Verified" badge for gitsign, so a private Rekor/Fulcio option is documented for enterprise.

12. Surfaces — CLI, SARIF, plugin, GitHub

CLI (vid)

Exactly seven subcommands; flags exist only on review.

  • vid report — the coverage report: per-unit state (reviewed / under-reviewed / stale / migration-needed / exempt / unreviewed) plus orphans, required-but-absent vouches, and unverified reviews.
  • vid status — a one-screen coverage pulse.
  • vid queue — the severity-triaged worklist of everything actionable (the review to-do list).
  • vid verify — the CI gate: exits non-zero on ORPHAN, exact-hash STALE, or an unmet scoped requirement.
  • vid review <target> — record a review of a unit at its current shape. --as <id> for a human, --model <api-name> for an AI model, --tool <name> for a 3rd-party tool. Defaults criteria/rigor; echoes the coverage effect.
  • vid json — emit the versioned vid.json coverage artifact to stdout.
  • vid sarif — emit per-file SARIF (the gating findings) to stdout.
Decision #1 — settledCI gate: ORPHAN, exact-hash STALE, or an unmet scoped requirement fail; the printed label is gate: orphan,stale,scope (N stale, N orphan, N requirement). Severity never gates (it only prioritizes the queue). A malformed/unreadable ledger fails closed.
Decision #4 — settledSARIF vs vid.json: split. SARIF emits the gating findings (4 rules: vid/stale error, warning when minor; vid/orphan error; vid/requirement error, gating; vid/under-reviewed note) for CI/code-scanning; coverage + lane + severity + each unit's missing shortfall live in a purpose-built vid.json that the plugin and vid report consume.

The two machine outputs & their versions

Both artifacts are versioned via embedded strings (there is no separate vid.json/SARIF JSON Schema file — schemas/ holds only the two record envelopes). vid.json carries a top-level schemaVersion = "vid.json/v2"; SARIF stays at version 2.1.0 and carries its vid contract separately in runs[0].properties.vidSchemaVersion = "vid-sarif/v1".

Editor integrations

Both the JetBrains / IntelliJ plugin (id com.vidivouch.vid) and a VS Code extension shell out to the vid binary and render the same vid.json coverage + per-file SARIF — implementation-language-agnostic, in lockstep. The "ViD" eye is the shared mark. The settled spine is a coverage sidebar / tool-window tree for the overview + inline inlay hints for in-flow provenance; the Vouch action lives on the code (the standardized Gutter Vouch gesture), never in a diagnostics/Problems list. Xcode joins as a Source Editor Extension (commands only — Xcode exposes no gutter/sidebar API). Detailed option mockups — a VS Code coverage webview, a commit-time gate, cross-editor "Review Mode" — live in editor-plugin-designs.html.

Editor decisions — settled 2026-06-19
  • Spine: sidebar/tool-window tree + inline inlay hints, both VS Code and JetBrains.
  • Gutter Vouch is the standard gesture — one action on the unit's status-coloured gutter marker (native click in JetBrains; a co-located CodeLens/hover command-link in VS Code, which has no gutter-click event; keybinding parity ⌃⌥V in Xcode). The marker is a status dot/bar, not the eye glyph.
  • No diagnostics dilution: stale units are never published to the Problems panel; the interaction stays on the code.
  • Commit-time Vouch is opt-in, pre-selecting nothing — a manual commit is not evidence of human review, so committing asserts no coverage unless a unit is deliberately ticked.
  • Plugins never edit source — every review is recorded to the .vid/ ledger, never injected as a code comment.
  • VS Code webview ships as an opt-in "Coverage" panel, to match the JetBrains tool-window's richness.
  • Review Mode (next → see provenance → Vouch) is a cross-editor standard surface.

Plugin settings — local presentation vs governed trust

The plugins expose settings split by blast radius. Review-state colours are local & live — per-editor presentation, with project defaults in committed .vid/config. Trust policy is governed: editing the axes, reviewers, scopes, thresholds, or exemptions does not apply locally — it opens a pull request against .vid/trust.toml (CODEOWNERS-gated, author-can't-approve-own-change), preserving the §10 TCB guarantees. The plugin shows the effective policy; it never silently rewrites it.

GitHub integration

vid's output is the "un-reviewed report," so the PR is where it lands. The primary PR surface is commitless: a reviewer vouches out-of-band — identity from their authenticated GitHub session, bound to the pull request's head commit — and the GitHub App posts a required Check-run. No commit is added to the branch, the diff under review is untouched, and the code CI is not re-triggered (the same model every check tool — Codecov, CodeQL, CodeRabbit — uses). The git-native append to .vid/reviews.jsonl remains the offline / single-actor tier (where the reviewer is also the committer, so "vouch = commit" is no role violation) — see VOUCH-ARCHITECTURE.html for why the hosted path is primary and the git-native path is the low-assurance fallback (and why a history-rewriting fixup is rejected). None of the planes edits your source.

The coverage matrix — the signature view — greenfield; designed to each medium's ceiling, not the lowest common denominator

The 2-axis policy makes coverage a spatial object, so the matrix is not a report appendix — it is the product's signature view, the way a flame graph is a profiler. Every requirement is a region (the accept rows trace a frontier — an AND is a corner, an OR is an L); every unit is a point at its strongest (human, machine); coverage is simply the mass that sits inside the frontier, and a gap is the distance a point must travel to get there. Build each surface to the best its medium allows; they share one source of truth (vid.json + the trust.toml axes), never one lowest-common-denominator rendering.

One model, rendered to each ceilingAll three read the same vid.json (each unit's state + its missing shortfall: scope → accept options → {axis, required, present} gaps) over the trust.toml axes. They are not the same picture down-rezzed — they are the best picture each medium can draw: a live SVG with a time-scrubber on the web, a navigable truecolor TUI (with inline graphics) in the terminal, and a code-linked panel with drag-to-vouch in the editor.

13. The un-reviewed report

Decision #3 — settledStratify by invariance grade (code/structural/opaque) and lane — the honesty floor — then score against the trust.toml accept-list (§10). Code-grade human-lane coverage is the headline; prose/markup/opaque grades and the machine lane (models + tools) are separate, distinctly-labeled — a machine review can never pad the human number, and an off-allowlist human review is recorded but never counted (surfaced as unverified).

Per-unit coverage states are reviewed / under-reviewed / stale / migration-needed / exempt / unreviewed, plus orphans and required-but-absent vouches. The accept-list is applied at report/verify time, so the bar is re-scorable without rewriting history. The headline says "code-grade units a verified human reviewed"; it does not claim understanding or correctness, and (phase 1) carries the unverified-identity banner.

14. Known limitations & residual blind spots

AcceptedExtract-method / cross-function move loses the review; mitigated by a non-authoritative "may have moved here" suggestion.
AcceptedProse weak-invariance — prose has no behavior, so any content change re-triggers; alpha-normalization disabled for prose, sentence-grained similarity, labeled structural/opaque so it never pads code coverage.
AcceptedFree-name rebinding — designed to be mitigated (not eliminated) by binding-kind sensitivity (§4); the v1 Rust profile does not yet implement it, so a local rename that shadows a free name reads STALE (the safe over-report direction). The hash attests to syntactic shape, not semantics.
AcceptedStolen interactive human session — an insider/agent with a live human OIDC session can sign as that human even in phase 2. Irreducible without per-review hardware attestation.

15. Versioning & migration

A profileVersion/scopeModelVersion/canonicalizer bump is an additive, auditable event, never a silent re-hash. Per-version hashers are retained; a version mismatch surfaces as the migration-needed state (advisory), never STALE/FAIL. Migration writes new positive records citing the old, so coverage is preserved monotonically and nothing silently flips reviewed → unreviewed.

16. Implementation & distribution (Rust)

Implementation language: Rust. Rationale: the ty / Ruff precedent (deep Rust static analysis of Python shipped as a single fast binary on PyPI + npm + brew); a single binary makes modern distribution near-free and kills the Python-freeze epic and the bundled-interpreter syntax ceiling; Ruff-class performance; and the foundational design above is implementation-agnostic, so it transfers unchanged. The Python probatum prototype is retained only as a reference implementation.

18. Build sequence & MVP scope

Greenfield Rust workspace, clean core first. The record envelope is the one irreversible decision; everything else is replaceable.

  1. Lock the record envelope — in-toto Statement (subject.digest={sha256} only; predicate per §7; DSSE-ready); the in-repo JSON Schemas in schemas/ are the source of truth (predicateType is an opaque id, its URL need not resolve).
  2. vid-core engine — node tree, exact hash (parameterized), true-multiset Ruzicka Jaccard, ordered/unordered driving both digests, severity = bucket(max(rel,abs)) + single-child floor. Tests: {s,s}≠{s}; reorder ⇒ STALE + nonzero severity; one-of-twenty ⇒ MINOR not NONE.
  3. NormalizationProfile trait + tree-sitter; first binding profile (Python or Rust) with its own CST scope binder; conditional class ordering; pinned scopeModelVersion (binding-kind sensitivity is a planned follow-up, not v1 — §4/§14).
  4. Monotonic store + verify — positive-claims-only union JSONL (reviews + REVOKEs), content-addressed dedupe, REVOKE (revoke-wins); policy is monotone by construction (absence is unrepresentable); treats the file as untrusted and fails closed on a malformed ledger. Test: deleting any line can only lower coverage.
  5. Move/rename reconcile — complexity floor + repo-global uniqueness + never-reuse-signature + one-keystroke re-confirm; empty-unit collision regression test.
  6. Signing seam scaffolding — ship the envelope, identity allowlist, issuer-audience separation, the unverified banner. Phase-1 lenient.
  7. trust.toml TCB — two unmerged axes, the [reviewers] allowlist, path-scoped accept lists, default, exemptions, severity thresholds; CODEOWNERS-gated; anti-self-promotion.
  8. vid CLI (clap) — the §12 surface.
  9. Add JS + Rust profiles (binding-language binders; self-hosting check).
  10. HTML + Markdown profiles (spec tree + injections; comrak container + fenced-code rung ladder).
  11. SARIF emitter (findings-only) + vid.json coverage artifact.
  12. Distribution — cargo-dist (npm/brew/installer) + maturin (PyPI wheel).
  13. JetBrains plugin — repoint the existing Kotlin plugin at the vid binary; com.vidivouch.vid; eye wordmark.
